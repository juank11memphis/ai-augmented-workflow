# Scrum Planning GitHub Issue Export Feature Brief

## Summary

After the Scrum planner skill creates Epics and User Stories, it should offer to create matching GitHub Issues when a GitHub MCP server is available. If the user declines, nothing else happens. If the user accepts, the skill creates fresh GitHub issues in the current repository: each Epic becomes a parent issue, each User Story becomes a child issue, and each User Story issue is attached to its Epic using GitHub's native sub-issue capability. This is a create-only convenience inside the planning workflow, not a sync or tracking system.

## Product Vision Fit

This feature supports Sibu's promise to turn AI-assisted development into a reliable loop instead of a pile of disconnected artifacts. Users already create planning docs through the Scrum planner skill; giving them an optional handoff to GitHub Issues reduces manual copy/paste and helps teams move from planning to execution while preserving user control.

The feature stays aligned with Sibu's philosophy because it is explicit and opt-in. The agent asks before touching GitHub, creates only what the user approved, leaves local planning docs unchanged, and does not pretend to own project management after export.

## Deep Module

- **Template Catalog and Rendering**: owns the Scrum planner skill template and manifest metadata that will teach agents when and how to offer GitHub issue export.
- **Workflow Target Planning**: provides selected MCP server context through the existing workflow model, but this feature should not change selectable MCP catalog behavior.
- **MCP Server Selection Management**: remains the owner of configuring GitHub MCP availability; this feature consumes that availability but does not change MCP lifecycle commands.

No new Deep Module is needed. The core product change is guidance inside the Scrum planner skill, with GitHub actions performed only when the agent has an available GitHub MCP server.

## User / Customer Problem

After Sibu helps a user create Epics and User Stories, the next practical step is often creating GitHub Issues so the work can be tracked and assigned. Doing this manually is repetitive and error-prone: users copy titles, acceptance criteria, paths, and parent/child relationships from local Markdown into GitHub.

Users need a lightweight bridge from Sibu's planning artifacts to GitHub Issues without turning Sibu into a project-management sync engine.

## Business Goal

Increase the usefulness of Sibu's planning pipeline by reducing friction between local AI-generated planning artifacts and the GitHub workflow many teams already use.

The feature should make the Scrum planner feel more complete: it produces high-quality local docs, then optionally helps users create the corresponding GitHub issue structure in one controlled step.

## Target User / Scenario

This is for solo engineers and teams using Sibu's Scrum planner skill in a GitHub-backed repository where a GitHub MCP server is available to the agent.

The target scenario is:

1. The user asks the Scrum planner skill to create Epics and User Stories from an approved feature brief and technical design.
2. The skill writes the local planning documents.
3. The skill detects that GitHub MCP capabilities are available.
4. The skill asks whether to create GitHub Issues for the generated Epics and User Stories.
5. The user chooses yes or no.

## Proposed Experience

After writing the local Epic and User Story files, the Scrum planner skill checks whether GitHub MCP access is available for the current repository.

If GitHub MCP is not available, the skill ends normally after reporting the created local planning docs.

If GitHub MCP is available, the skill asks a concise opt-in question such as:

> GitHub MCP is available. Create GitHub Issues for these Epics and User Stories in the current repo?

If the user says no, the skill does nothing else.

If the user says yes, the skill creates issues in the current local repository only:

- each Epic becomes a GitHub Issue
- each User Story becomes a GitHub Issue
- each User Story issue is attached as a native GitHub sub-issue of its parent Epic issue
- Epic issues receive the labels `epic` and `sibu-generated`
- User Story issues receive the labels `user-story` and `sibu-generated`
- missing labels are created automatically
- local Markdown files are not modified

The skill reports what it created and clearly reports any failure.

## GitHub Issue Mapping

### Epic issue

Each Epic document maps to one parent GitHub Issue.

Recommended title format:

```txt
Epic: <Epic Title>
```

Recommended body content:

- Epic summary or goal
- Scope at a business/planning level
- Links or references to local source doc paths
- A note that the issue was generated by Sibu

Labels:

- `epic`
- `sibu-generated`

### User Story issue

Each User Story document maps to one GitHub Issue.

Recommended title format:

```txt
Story: <Story Title>
```

Recommended body content:

- User story statement
- Acceptance criteria
- Relevant notes or constraints from the story document
- Parent Epic reference
- Local source doc path
- A note that the issue was generated by Sibu

Labels:

- `user-story`
- `sibu-generated`

### Parent / child relationship

The MVP must use GitHub's native sub-issue capability. User Story issues should be attached as sub-issues of their parent Epic issue.

If native sub-issue creation fails or is unavailable, the MVP should fail clearly. It should not silently fall back to Markdown checklists, loose links, GitHub Projects, milestones, or another relationship model.

## MVP Scope

- Update the Scrum planner skill guidance so that after creating Epics and User Stories it checks for GitHub MCP availability.
- Ask the user whether to create GitHub Issues only when GitHub MCP is available.
- Do nothing if the user declines.
- Create issues only in the current local repository.
- Create a fresh set of GitHub issues every time the user confirms; do not detect duplicates.
- Create one parent issue for each Epic.
- Create one issue for each User Story.
- Attach each User Story issue as a native GitHub sub-issue of its parent Epic issue.
- Create the labels `epic`, `user-story`, and `sibu-generated` when missing.
- Apply `epic` and `sibu-generated` to Epic issues.
- Apply `user-story` and `sibu-generated` to User Story issues.
- Leave all local Markdown planning files unchanged.
- Report created issue URLs or numbers after successful export.
- Fail clearly if GitHub issue creation, label creation, or native sub-issue attachment fails.

## Out of Scope

- Adding a new `sibu` CLI command for GitHub issue export.
- Manually invoking a separate Sibu command after Scrum planning.
- Updating existing GitHub issues.
- Detecting duplicate issues.
- Searching for previously exported issues.
- Syncing GitHub issue status back to local Markdown files.
- Tracking issue state, assignment, labels, milestones, projects, or completion over time.
- Modifying local Epic or User Story Markdown files with GitHub URLs.
- Creating issues in any repository other than the current local repo.
- Letting users choose a different GitHub repository during export.
- Falling back to Markdown checklists or links when native sub-issues are unavailable.
- GitHub Projects integration.
- Milestone creation.
- Assignee management.
- Closing or reopening issues.
- Managing credentials, GitHub authentication, or MCP server setup.

## Success Signals

- Users who generate Epics and User Stories can immediately create matching GitHub Issues when GitHub MCP is available.
- The opt-in prompt makes GitHub mutation explicit and easy to decline.
- Created GitHub Issues preserve the planning hierarchy through native sub-issues.
- Users do not have to copy/paste planning docs into GitHub manually.
- Local planning files remain unchanged after export.
- Failures are visible and understandable instead of producing partial silent behavior.

## Business-Level Acceptance Criteria

- Given the Scrum planner skill has created Epic and User Story files, when GitHub MCP is unavailable, then the skill does not offer GitHub issue export and ends normally.
- Given GitHub MCP is available, when the Scrum planner finishes creating local planning docs, then it asks whether to create GitHub Issues in the current repository.
- Given the user declines, then no GitHub labels, issues, sub-issues, or local file changes are made.
- Given the user accepts, then the skill creates any missing labels among `epic`, `user-story`, and `sibu-generated`.
- Given the user accepts, then each Epic becomes a parent GitHub Issue with `epic` and `sibu-generated` labels.
- Given the user accepts, then each User Story becomes a GitHub Issue with `user-story` and `sibu-generated` labels.
- Given the user accepts, then each User Story issue is attached as a native sub-issue of its parent Epic issue.
- Given native sub-issue attachment fails, then the export fails clearly and does not claim success.
- Given export completes, then the skill reports the created GitHub issue numbers or URLs.
- Given export completes, then local Epic and User Story Markdown files remain unchanged.
- Given the same Scrum planning artifacts are exported more than once, then the MVP creates a fresh issue set each time after user confirmation.

## Risks / Tradeoffs

- Always creating fresh issues can produce duplicates if users confirm export multiple times. This is intentional for MVP simplicity but should be explained by the prompt or summary.
- Native sub-issues may not be available in every GitHub context or MCP capability set, so the feature must fail clearly instead of degrading into a weaker relationship model.
- GitHub MCP tooling may not expose every needed operation directly; implementation may need a careful technical design to confirm how issue creation, label creation, and sub-issue attachment will be performed.
- Adding GitHub mutation to a planning skill increases responsibility; the opt-in prompt must remain explicit and safe.
- Label creation may conflict with existing repository label conventions, but the MVP uses simple, low-surprise labels.
